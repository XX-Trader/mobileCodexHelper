# 聊天体验升级计划

## 1. 背景

当前项目的聊天体验已经可用，但在会话状态提示、未读提醒、阅读定位、输入区记忆、多会话切换、主题切换和通知能力上仍有明显缺口。

这次升级目标不是做零散 UI 修补，而是统一整理一套“会话状态 + 未读提醒 + 输入记忆 + 快速切换 + 主题/通知”的交互体系，优先保证：

1. 可调试性
2. 状态一致性
3. 手机端与电脑端体验一致
4. 尽量复用现有前端结构，不引入新第三方依赖

## 2. 目标

完成后应达到以下效果：

1. 新建或处理中会话能稳定显示绿色状态。
2. 已回复但未查看的会话能稳定显示红色未读状态。
3. 项目级显示数字，会话级不显示数字，只显示状态。
4. 打开未读会话时能直接定位到合理位置。
5. 每个会话都能记住自己的未发送输入内容和输入模式。
6. 输入区支持继续插入内容，支持插入图片。
7. 多会话切换足够快，适合手机端使用。
8. 支持一键切换夜间模式。
9. 支持浏览器通知，且可以手动开关提醒。
10. 聊天中修改后的文件链接可直接打开，尤其支持 `.md` 文件查看。
11. 多个仍在进行中的会话之间切换时，聊天内容能稳定加载。

## 3. 已确认需求

### 3.1 状态与数字显示

1. 会话级别不显示数字，只显示状态。
2. 项目级别显示数字，值为 `0` 时不显示。
3. 项目级允许同时存在两个数字，但要分区显示，不能混在一起。
4. 长项目名不能遮挡状态数字。
5. 会话前方保留绿色点或红色状态提示。

### 3.2 阅读与未读

1. 红色表示“已经回复完成，但我还没有看”。
2. 浏览器通知只针对红色未读会话。
3. 已读/未读判断必须稳定，当前“回答完毕后有时不变红”的问题要专项排查。

### 3.3 输入区与草稿

1. 每个会话的输入框内容跟随会话保存。
2. 每个会话记住输入模式。
3. 允许继续插入信息，适合补充遗漏内容。
4. 支持插入图片。
5. 希望前端本地保存，提高切换效率。

### 3.4 多会话切换

1. 需要快速切换多个聊天。
2. 可以增加提醒开关。
3. 方案以手机端使用方便为优先。

### 3.5 主题与显示

1. 支持一键夜间模式。
2. `edit` 文件编辑字体默认增大一号。
3. 用户发送的长消息支持折叠。

### 3.6 文件链接与 Markdown 查看

1. 修改后的文件如果在聊天里有链接，希望点击后能直接查看。
2. `.md` 文件需要完整支持查看，最好能直接进入 Markdown 预览/编辑链路。
3. 前端和后端都要补齐，不只做前端展示。

### 3.7 多活动会话切换稳定性

1. 多个会话同时在进行中时，切换到另一个会话，聊天内容有时没有加载出来。
2. 需要定位根因。
3. 需要纳入计划并彻底修复，而不是只做表面刷新按钮。

## 4. 推荐交互方案

### 4.1 会话级与项目级状态拆分

#### 会话级

只显示状态，不显示数字：

1. 绿色点：会话正在处理中或正在回复。
2. 红色点：会话已有未查看回复。
3. 无状态：没有处理中，也没有未读。

#### 项目级

项目右侧使用固定宽度状态区，避免被长项目名挤压：

1. 绿色数字：该项目下“正在处理”的会话数量。
2. 红色数字：该项目下“已回复但未查看”的会话数量。
3. 值为 `0` 时不显示。
4. 两种数字分开显示，不共用一个 badge。

#### 设计取舍

项目显示数字、会话只显示状态，能在列表层级上快速看出“哪个项目有多少待关注会话”，同时避免会话行信息过载。

### 4.2 未读与阅读定位统一规则

推荐规则如下：

1. 未读会话首次打开时，优先定位到“我最后一条消息”附近。
2. 同时高亮未读起点，帮助快速识别从哪里开始看。
3. 已读会话默认恢复该会话上一次滚动位置。
4. 如果没有可恢复滚动位置，则回到底部。

#### 为什么不能只靠“记住滚动位置”

如果只恢复滚动位置，未读会话可能仍停留在旧位置，用户看不到新回复。  
因此建议优先级采用：

`未读定位 > 已保存滚动位置 > 默认到底部`

### 4.3 输入区会话记忆

每个会话独立保存以下内容：

1. 未发送文本草稿
2. 输入模式
3. 滚动位置
4. 输入框是否展开

图片草稿建议区分两层：

1. 基础版：当前页面生命周期内保留图片草稿。
2. 增强版：跨刷新保存图片草稿，使用 IndexedDB。

#### 推荐实现

1. 第一阶段先落地文本草稿、输入模式、滚动位置。
2. 图片插入先支持当前会话内即时插入与发送。
3. 图片草稿跨刷新持久化作为增强项单独实现。

原因：

1. `File` 对象不能直接可靠保存到 `localStorage`。
2. 直接把图片持久化放进首批改动，会显著增加复杂度和调试成本。

### 4.4 输入区插入能力

新增“插入”能力，目标是补充信息更顺手，而不是重新编辑整段大文本。

建议提供的插入能力：

1. 插入图片
2. 插入当前时间
3. 插入项目路径
4. 插入最近一次用户消息内容
5. 插入常用模板片段

实现方式建议为：

1. 输入框附近增加“插入”按钮
2. 点击后打开轻量菜单
3. 插入内容追加到当前光标位置，而不是覆盖整段文本

### 4.5 浏览器通知

浏览器通知只针对：

1. 会话已完成回复
2. 当前仍未查看
3. 且提醒开关已开启

通知策略：

1. 当前正在看的会话不通知
2. 已读会话不通知
3. 重复事件要去重，避免同一条回复弹多次
4. 提供全局开关，允许关闭提醒

#### 已知限制

1. 桌面浏览器通常支持较好。
2. 部分手机浏览器，尤其 iOS Safari，对浏览器通知支持有限。
3. 因此浏览器通知必须是增强能力，不能替代站内红色未读状态。

### 4.6 多会话快速切换

推荐采用“最近会话条 + 原侧边栏”的组合方案：

1. 主聊天区顶部增加最近会话条，展示最近 5 个会话。
2. 点击可快速切换。
3. 侧边栏继续保留完整会话树。
4. 提醒开关放在全局设置区或标题栏下拉中。

原因：

1. 比多窗口/分屏简单，手机端更可控。
2. 切换路径更短。
3. 不破坏现有侧边栏结构。

### 4.7 夜间模式

增加全局夜间模式开关：

1. 一键切换浅色/深色
2. 跟随本地存储记忆
3. 覆盖聊天区、侧边栏、编辑器和弹层

优先保证：

1. 文本可读性
2. 状态颜色在深色背景下仍有足够对比度

### 4.8 用户长消息折叠

针对用户自己发送的长消息：

1. 超过指定高度或字符数时默认折叠
2. 提供展开/收起按钮
3. 默认保留前几行预览

建议阈值：

1. 大于 `8` 行
2. 或大于 `280` 字符

### 4.9 编辑器字号

编辑器默认字体从当前值上调一档：

1. 默认字号从 `12` 提升到 `13`
2. 仅影响未自定义过字号的用户
3. 已自定义字号用户保持原设置

### 4.10 文件链接与 Markdown 打开闭环

推荐把“聊天消息里的文件链接”做成完整闭环，而不是只依赖已有文件树：

1. 聊天中的 `Edit`、`Write`、`ApplyPatch` 结果里，文件链接可直接点击。
2. 点击后统一走 `onFileOpen` 打开侧边编辑器。
3. `.md` 文件优先支持 Markdown 预览与编辑切换。
4. 后端 `readFile` / `saveFile` / 文件存在性检查链路需要一起校验，确保 Markdown 文件和普通文本文件都能稳定打开。
5. Markdown 预览里的本地相对链接要基于当前文件目录解析，并统一跳转到 `/file-preview`，不能直接交给浏览器按当前路由拼接。

设计取舍：

1. 前端其实已经具备 Markdown 预览能力。
2. 真正需要补齐的是“聊天链接 -> 文件打开 -> 后端读取 -> 编辑器/预览展示”的完整链路。
3. 这样对查看计划文档、PRD、README、变更说明最友好。

## 5. 技术实现建议

### 5.1 状态链路修复

优先检查并统一以下链路：

1. `active-sessions`
2. `session-notifications-state`
3. `acknowledgeSession`
4. 临时新会话 `new-session-*`
5. 新建会话转真实会话 ID 的替换逻辑

当前风险判断：

1. 现有前端 override 与上游版本有分叉。
2. 已有会话状态提示逻辑很可能被旧 override 覆盖。
3. “新会话无绿点”和“完成后不变红”大概率都与状态同步链不完整有关。

### 5.2 本地存储策略

建议按会话维度设计前端存储键：

1. `draft_input::<provider>::<sessionId>`
2. `draft_mode::<provider>::<sessionId>`
3. `draft_scroll::<provider>::<sessionId>`
4. `draft_ui_state::<provider>::<sessionId>`

图片增强版可使用：

1. IndexedDB：存图片草稿 Blob
2. `localStorage`：只存图片草稿索引

### 5.3 通知策略

浏览器通知触发条件：

1. 会话进入未读状态
2. 通知开关开启
3. 页面非当前会话
4. 本次未读事件未通知过

去重键建议：

1. `provider + sessionId + latestMessageId`

### 5.4 文件链接与 Markdown 查看链路

建议重点检查：

1. 聊天工具渲染器里对文件链接的点击动作是否统一调用 `onFileOpen`
2. 编辑器加载文件时是否统一走 `api.readFile`
3. 后端对 `.md` 文件是否和普通文本文件一样允许读取
4. Markdown 文件打开后是否能直接进入预览/编辑模式
5. Markdown 预览中的相对链接是否会在新窗口里打开到正确文件，而不是错误的前端路由地址

当前判断：

1. 前端已有 Markdown 预览能力。
2. 补齐后端读取与打开链路，收益会比单独做一个“查看 md”按钮更高。

### 5.5 多活动会话切换未加载问题定位

当前高概率根因有两类，需要一并排查：

1. `useProjectsState` 在存在活动会话时，为了保护当前会话稳定，会在部分 `projects_updated` 事件上直接跳过 `setProjects(updatedProjects)`。
2. `useChatSessionState` 对会话加载又有基于 `sessionKey` 的跳过逻辑，如果前一层状态没有及时同步，切换到另一个仍在进行中的会话时，容易出现“UI 已切换，但消息没有重新拉取”。

后续联调中又确认了三类更具体的滚动与加载风险，需要写入同一专项：

1. 会话加载 effect 如果依赖 `chatMessages.length`、`sessionMessages.length` 或整个 `selectedProject` 对象，会在同一会话内被消息变化反复触发，造成重复 `load-session`、DOM 重建和滚动抖动。
2. 连续 Bash / 命令工具消息如果在渲染阶段被重新 regroup 成摘要块，会导致消息区在运行中出现“完整命令块 <-> 摘要块”结构切换，直接改变列表高度。
3. 聊天容器在长回复或长命令期间持续增高时，浏览器原生 scroll anchoring 可能与“保持底部跟随”逻辑冲突，表现为滚动条自己上下跑。

从现有代码看，风险点集中在：

1. 活动会话保护条件过宽
2. 背景活动会话的外部消息刷新只在“当前选中的会话且不处于 active”时才触发
3. 多活动会话并行时，前端状态机对“当前视图会话”和“后台活动会话”的边界不够清晰
4. 会话加载链依赖项过宽，导致同一会话内重复 reload
5. 流式命令消息在渲染层被二次重排
6. 容器高度变化与滚动锚定策略没有统一

修复原则：

1. 保护当前可见会话的流式渲染
2. 不阻断其他活动会话的元数据同步
3. 切换会话时强制保证目标会话消息重新校验与拉取
4. 同一会话的消息加载 effect 只依赖稳定的 `project + provider + sessionId` 键，不依赖消息长度
5. 流式命令消息在回答进行中保持结构稳定，不做运行中 regroup
6. 聊天容器禁用原生 `overflow-anchor`，并在“用户原本就在底部”时由前端统一执行贴底跟随

### 5.6 聊天滚动稳定性专项

这一项在实施后需要单独验收，不能只作为“未读定位”的附带效果。

当前确认的稳定性规则如下：

1. 滚动只有一个真相源，所有主动改 `scrollTop` 的入口都要可追踪。
2. 未读定位、已保存滚动恢复、默认到底部的优先级仍保持：
   `未读定位 > 已保存滚动位置 > 默认到底部`
3. 用户手动上滑后，不允许因为容器高度变化被强制拉回底部。
4. 用户原本就在底部时，长回复、长命令、状态条变化引起的容器高度增长，应保持底部跟随。
5. 顶部最近会话条、底部状态条、命令区摘要等辅助 UI 不能在流式过程中突然插入/移除，必须尽量保留稳定占位。
6. 聊天中的连续命令消息宁可保持逐条渲染，也不要在运行中为了“更紧凑”而切换成另一种结构。
7. 顶部最近会话条支持“仅移除快捷入口、不删除真实会话”，并复用已有未读红点、处理中绿点和浏览器提醒开关链路。
8. 顶部最近会话条采用固定槽位顺序，某个会话进入顶部后的相对位置保持不变，切换会话只改变高亮态，不按“最近点击”重新排序。

## 6. 分阶段实施计划

### 阶段 1：状态系统修复

目标：

1. 修复绿色处理中状态
2. 修复红色未读状态
3. 项目级数字与会话级状态分离
4. 修复长项目名遮挡 badge

预计改动文件：

1. `upstream-overrides/claudecodeui-1.25.2/src/components/app/AppContent.tsx`
2. `upstream-overrides/claudecodeui-1.25.2/src/components/sidebar/view/Sidebar.tsx`
3. `upstream-overrides/claudecodeui-1.25.2/src/components/sidebar/view/subcomponents/SidebarProjectItem.tsx`
4. `upstream-overrides/claudecodeui-1.25.2/src/components/sidebar/view/subcomponents/SidebarSessionItem.tsx`
5. 相关会话状态 hook

验证点：

1. 新建会话发送后立即出现绿色状态
2. 回复结束后未查看时出现红色未读状态
3. 打开后未读清除
4. 项目级绿色数字和红色数字分开显示
5. 长项目名不再遮挡数字

### 阶段 2：通知与提醒开关

目标：

1. 增加浏览器通知
2. 增加提醒总开关
3. 仅对红色未读会话通知

预计改动文件：

1. 应用层状态管理
2. 设置或标题栏相关组件
3. 本地通知管理工具模块

验证点：

1. 红色未读会话能收到通知
2. 当前会话不弹通知
3. 关闭开关后不再通知
4. 重复回复不重复通知

### 阶段 3：文件链接与 Markdown 查看链路补齐

目标：

1. 聊天中的文件链接可直接打开
2. `.md` 文件可稳定查看
3. Markdown 文件支持预览/编辑切换
4. 前后端文件打开链路统一

预计改动文件：

1. 聊天工具渲染与文件点击相关组件
2. 主内容文件打开链路
3. 编辑器文件加载模块
4. 后端文件读取相关路由或适配层

验证点：

1. 点击聊天里的修改文件链接能直接打开文件
2. `.md` 文件能正常加载内容
3. Markdown 预览正常显示
4. 文件不存在或读取失败时能给出明确错误
### 阶段 4：输入区会话记忆与插入能力

目标：

1. 每会话记住未发送文本
2. 每会话记住输入模式
3. 每会话记住滚动位置
4. 支持插入内容
5. 支持插入图片

预计改动文件：

1. 聊天输入区组件
2. 聊天输入状态 hook
3. 本地存储工具模块
4. 图片上传与预览相关组件

验证点：

1. 多会话切换后草稿不串
2. 切回原会话还能看到之前输入的文字
3. 输入模式随会话恢复
4. 可在原文本中继续插入内容
5. 图片可插入并发送

### 阶段 5：阅读定位与消息折叠

目标：

1. 未读会话自动跳到我最后一条消息附近
2. 已读会话恢复滚动位置
3. 用户长消息支持折叠
4. 长回复/长命令期间聊天页不再明显上下跳动

预计改动文件：

1. 聊天消息状态 hook
2. 消息列表组件
3. 单条消息组件
4. 滚动调试与滚动跟随相关逻辑

验证点：

1. 未读会话打开位置合理
2. 已读会话返回时保留阅读位置
3. 长消息默认折叠，可展开/收起
4. 长命令执行时滚动条不会因列表重排而明显上下跳动

### 阶段 6：快速切换、会话切换稳定性与夜间模式

目标：

1. 增加最近会话条
2. 提升多会话切换效率
3. 修复多个活动会话切换后内容未加载的问题
4. 增加一键夜间模式
5. 编辑器默认字号加一号

预计改动文件：

1. 主内容头部组件
2. 会话切换相关组件
3. 项目/会话状态管理 hook
4. 聊天会话加载 hook
5. 全局主题状态与样式
6. 编辑器设置模块

验证点：

1. 最近会话条可快速切换
2. 多个活动会话之间切换时，目标会话内容稳定加载
3. 夜间模式一键切换生效
4. 刷新后夜间模式保留
5. 编辑器字体明显变大

## 7. 阶段依赖关系

1. 阶段 1 必须先做，因为后续通知、未读跳转、项目级统计都依赖正确的状态链。
2. 阶段 2 依赖阶段 1 的未读状态稳定。
3. 阶段 3 建议早做，因为计划文档和 Markdown 文件查看会直接受益。
4. 阶段 4 与阶段 5 可并行设计，但建议先做阶段 4，再做阶段 5。
5. 阶段 6 放最后，便于在状态链、输入记忆、文件打开链路稳定后统一收口。

## 8. 风险与已知限制

1. 前端 override 与上游版本有分叉，合并时容易引入回归。
2. 浏览器通知在不同手机浏览器上的支持能力不一致。
3. 图片草稿跨刷新持久化复杂度高于文本草稿。
4. 滚动位置恢复与未读跳转存在优先级冲突，必须统一规则。
5. 文件链接链路牵涉前端跳转与后端文件读取，联调时要重点验证 Markdown、中文路径和不存在文件三类场景。
6. 多活动会话切换问题本质上涉及状态同步策略，修复时要避免破坏当前正在流式输出的会话。
7. 聊天滚动稳定性不仅受 `scrollTop` 逻辑影响，还受渲染结构切换和浏览器原生滚动锚定影响，排障时必须同时观察容器高度变化。

## 9. 不引入第三方依赖原则

本计划默认不新增第三方依赖，优先使用：

1. Web Notification API
2. `localStorage`
3. IndexedDB
4. 现有 React 状态与项目内工具函数

只有在以下情况才考虑新增依赖：

1. IndexedDB 封装复杂度过高
2. 浏览器兼容性处理成本明显高于轻量库

若后续确实需要引入依赖，需单独评估维护状态、安全性和包体积影响。

## 10. 验收标准

全部完成后应满足：

1. 新建会话、处理中会话、未读会话状态稳定无误。
2. 项目级数字与会话级状态清晰分离。
3. 浏览器提醒可开关，仅对红色未读会话生效。
4. 聊天中的文件链接，尤其是 `.md` 文件，可稳定打开和查看。
5. 每个会话的输入框内容、输入模式、滚动位置独立保存。
6. 输入区支持插入文本与图片。
7. 未读会话阅读定位合理。
8. 多活动会话切换时内容稳定加载。
9. 多会话切换效率明显提升。
10. 夜间模式和编辑器字号调整可用。
11. 同一会话不会因为前端状态变化而反复 reload 消息。
12. 长命令、长回复、状态条刷新期间，聊天页不会出现明显的上下跳动。

## 11. 当前待确认项

当前仍保留一个实现层面的待确认项：

1. 图片草稿是否需要“跨页面刷新仍保留”。

推荐答案：

1. 第一版先不做跨刷新图片草稿持久化，只做文本/模式/滚动位置持久化。
2. 图片先支持会话内插入与发送。
3. 后续若你确实高频需要，再单独补 IndexedDB 持久化。

这个取舍能显著降低第一轮改动风险。
